초기 프로젝트파일에서는 엔터티 생성과 세션기반 로그인만 구현했으나 JWT를 사용한 로그인방식으로 변경함 
기존에는 세션기반으로 로그인 상태유지를 해왔으나 현재는 매 페이지마다 Common.js의 CheckLogin api 가 호출됨 이 함수에서는 로컬스토리지에 액세스토큰을 불러와 헤더에 authorization 정보를 추가해줌 그다음 해당 api를 위한 컨트롤러에서는 jwtUtil이라는 컴포넌트 클래스 파일을 만들어 검증,생성,리프레시토큰,id추출과 같은 메소드들이 들어가있기에 헤더정보에서 토큰값을 추출하여 로그인 상태가 유효한지 체크하고있음 지식의 한계인지 본문 자체에 header를 추가하는게 안되고있음.

카카오로 로그인 버튼을 누르면 호출되는 메소드 KakaoLogin, KAKAO_AUTH_URL 과 클라이언트_아이디,리다이렉트uri를 제공받아 카카오서버로 리다이렉션 url을 생성해줌 ,.인증이되었다면 /oauth/callback이라는 엔드포인트로 카카오가 이동을 시키는데 여기에서는 코드 파라미터가 주소창에 추가가된걸 확인할수있음 
이 code로 무얼하나 봤더니 카카오 액세스토큰에 접근할수있는 코드였음 , 허나 나는 기존에 세션기반 로그인 체제를 지웠기에 카카오를 통한 인증으로 어떻게 로그인 상태를 유지할지 고민이었음, 처음에는 CheckLogin 이라는 Api를 공통으로 사용하고싶기도하고 다른 스크립트의 추가나 코드가 추가되는것이 과연 맞을까라는 의구심이 생겼음
생각보다 간단했음 jwt에서도 그렇고 토큰에는 JSON형태의 마치 주민등록증같은  정보들이 저장되는데 거기서 userId나 이름과 같은 정보들을 불러와진다는 것에서  힌트를 얻어 Jsonparser를 이용하여 값이 id인부분의 value만 문자열로 뽑아올수있었음 
그 id값을 이용하여 기존 JWT도입시에 생성해두었던 jWT토큰 발급 로직을 적용시켜보았고 성공적으로 토큰발급이 완료되었음 , 그 토큰을 기반으로 CheckLogin API에서 공통으로 사용할수있었고 로그인 상태 유효성도 검증 할수있었음,
