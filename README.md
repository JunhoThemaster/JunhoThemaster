1. 기존에는 flight(회차)와 aircraft(항공기) 는 일대일 관계였다 하지만 상품 추가 시에 최초회차 설정시에 비행기를 따로 추가해주지 않으면 이미 사용중인 비행기는 사용하지 못한다는 것이었다.. 이를 유동적으로 바꾸기위해
   다대다 관계로 변경했고  flight 와 aircraft 사이의 flightsinAircraft라는 조인테이블을 만들어주었더니 비로소 aircraft 1개의 객체가 여러 회차를 가질수있었다. 따라서 10월 31일자 flight(회차)에 에 배정된 (aircraft)비행기가 
   11월 20일 자 flght(회차)에서도 사용이가능하고 예약할 seat들도 seat의 resrvedDate에 flight의 출발일자 와 같게 만들어주어 날짜만 겹치지 않는다면 중복사용 가능하게 만들어주었다.

   클라이언트에서의 응답데이터기 id:7}id:7}}id:7}}id:7}}id:7}}}}}}}}} ... 이런식으로 나오게 되었다.. 일단 임시방편으로 @JsonIgnore 어노테이션으로 해결하였지만 이게 정석적인 방법은아닌거같다.



2. 기존 common.js에서의 checkLogin함수는 로그인 검증 과 레이아웃 변경을 수행해주었는데, 어드민페이지나 권한이 필요한 페이지 이동시에 새로운 인증 api를 만들필요가 없다 생각하여 checkLogin 에서의 boolean값을 반환하게 만들어주었다
   후에 accessToAdmin이라는 스크립트에서는 checkLogin을 다시 호출하여 admin인지 아닌지 LoggedIn 인지 아닌지 검증후 ajax으 요청이아닌 url로 바로 보내주었다. url은 admin페이지 전체가 /admin으로 시작한다는 공통점을 가지기에 버튼 클릭시에 url을 직접 지정해주어
   각각 다른 버튼이어도 하나의 스크립트만으로 동작을 수행가능하게 만들었다.
