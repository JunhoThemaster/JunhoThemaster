트러블 슈팅 그리고 발전.

1.JPA를 통한 엔터티 생성후 참조관계설정시에 문제, 객체 정보를 뽑아올때의 필요없는 컬럼도 같이 불러오려해 문제
  Flight(회차) 기한 만료된 회차를 삭제시에 연결되어있던 aircraft와 seat들이 한꺼번에 삭제되는 문제.

 
 
해결방안: @PreRemove 어노테이션을 사용하여 해당 회차가 삭제되기전 연결되어있던 aircraft만 참조관계를 제거해주면 되지 않을까 싶어
  해보니 성공하였음, 또한 Flight 객체와의 상호작용을위해 JSON타입으로 직력화시 자식 테이블인 Aircraft도 불러오려해 오류 발생..@JsonIgnore어노테이션으로 해결


2. 세션기반 로그인의 코드의 중복성 addMemberToModel이라는 메소드를 따로 만들어서 중복되는 코드를 줄이긴했지만 여전히 각 엔드포인트를 담당하는 api 컨트롤러에서 다 호출해줘야한다는 번거로움.



해결방안: JWT 인증 방식 로그인으로 바꿨고 토큰발급후 매 페이지 호출시 토큰의 유효성을 정적 엔드포인트에서 단일적으로 검사.


3. JWT 인증 방식의 도입은 좋았으나헤더정보를 본문에 추가하는것이 안됐음...그래서 각 페이지의 엔드포인트 컨트롤러에서 헤더정보를 요청하여도 반환할수없었기에 로그인 상태의 검증이 불가함.



해결방안: 로그인시에는 로그인 스크립트에서는 id와 pw값을 보내며 로그인요청을 하게되고 서버에서는 토큰을 발급하고 클라이언트로 뿌려줌 만약 응답이 정상적이라면 토큰을 로컬스토리지에 저장하였었음.
이렇게 했던점을 참고하여 토큰값을 set이아닌 get으로 다시가져올수있었음 , 그걸 기반으로 헤더 정보에 토큰값을 넣어 보낼수있었고 서버측 CheckLogin api도 정상적으로 헤더정보를 읽기시작함.

4. JWT 도입 경험을 토대로 Oauth2.0을 이용한 소셜로그인 기능을 추가하려했고  카카오 디벨로퍼에서 클라이언트키를 받고 리다이렉트uri를 설정하여 카카오 로그인시 카카오로그인 페이지 호출에 성공함 ,callback엔드포인트에서 url에 code라는 파라미터로 카카오 서버의 토큰을 가져올수
  있었음 허나 그 토큰을 이제 어떻게 다루냐 생각하기전에 문제가 있었음  code파라미터가 포함된 callback엔드포인트에서는 어떠한 내 프로젝트의 공통 레이아웃을 참조하고있지 않기에 앞서만든 CheckLogin api가
  작동하지 않을 뿐더러 html본문에는 카카오서버 토큰의 값과같은 것들의 JSON형태만 반환이 되었음 이로인해 로그인 검증은 물론이며 카카오로 로그인시에는 다른페이지로 로드되지않는 문제가 발생함.    


 
  
해결방안: 토큰발급도 성공했고 토큰을 파싱하는 과정만 남았는데 토큰의 정보를 서버로 전달해줄 매개쳬가 필요했음.. 그래서 중간에서 데이터를 가로채줄수 있게 none.html이라는 아무것도 없는 html 파일을 만든후 
  그 html 본문에다가 스크립트를 작성함, 이 페이지가 로드가 됨과 동시에 실행되는 스크립트인데 api/session-info라는 api를 만들어주었고 그전에  callback엔드포인트에서의 컨트롤러에서는
  AccessTokenK라는 카카오에서 발급받은 토큰을 파싱하여 id를 추출후 내가 만든 jwtUtil에서의 토큰발급 로직을 적용함 그 토큰을 cookie에설정해두었고 httpOnly속성을 true로 하여 스크립트 수정을 불가하게만듬 이후에 none.html을 리다이렉팅 시키고 api/session-info 의 api에서 쿠키정보를 뽑아 토큰을 
  다시 클라이언트 측으로 뿌려 로컬스토리지에 토큰을 저장후 / 로 이동시키니 CheckLogin API가 카카오든 로컬 사용자든 공통적으로 정상적으로 작동함.
